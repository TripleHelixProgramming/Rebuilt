## Methodology

### Purpose

This analysis compares the vision processing behavior between:
1. **Original code** - The behavior recorded during actual robot operation
2. **New code** - The behavior produced by replaying the same inputs through the updated `VisionFilter` implementation

### Tools

A Python script (`scripts/compare_vision_logs.py`) was developed to parse WPILOG binary files and extract vision-related telemetry. The script:

1. Parses both log files using WPILib's DataLog format specification
2. Extracts all entries containing "Vision" in the key name
3. Compares `/RealOutputs/` (original behavior) vs `/ReplayOutputs/` (new behavior) within the simulation log
4. Computes statistics and generates distribution comparisons

### Data Sources

AdvantageKit replay logs contain two sets of outputs:
- **RealOutputs**: The outputs recorded during the original run (preserved from the input log)
- **ReplayOutputs**: The outputs generated by running the new code against the same inputs

This allows direct comparison of how the same vision observations are processed by different code versions.

### Key Metrics Analyzed

| Metric | Source | Description |
|--------|--------|-------------|
| `Vision/Summary/ObservationScore` | Output | Final score assigned to each accepted observation |
| `Vision/Summary/RobotPosesAccepted` | Output | Poses that passed the minScore threshold |
| `Vision/Camera{N}/PoseObservations` | Input | Raw pose observations from each camera |
| `Vision/Camera{N}/TagIds` | Input | AprilTag IDs detected by each camera |

## Analysis

### Key Finding: Fundamentally Different Scoring Behavior

The original code produced scores clustered in a narrow band just above the minimum threshold (0.02-0.10). This indicates the original scoring mechanism:
- Applied binary pass/fail logic, not continuous scoring
- Assigned similar scores to all "passing" observations regardless of quality
- Did not differentiate between excellent and marginal observations

The new weighted geometric mean approach produces a meaningful distribution:
- **Low scores (0.0-0.1)**: Only 55 observations (0.7%), representing genuinely poor data
- **Medium scores (0.5-0.7)**: 2,201 observations (27%), acceptable but not ideal
- **High scores (0.7-0.9)**: 5,202 observations (63%), good quality observations
- **Excellent scores (0.9-1.0)**: 544 observations (7%), highest confidence

### Impact on Pose Estimation

The score directly affects standard deviation calculations:

```java
double linearStdDev = linearStdDevBaseline / score;
double angularStdDev = angularStdDevBaseline / score;
```

| Score | Linear StdDev (m) | Angular StdDev (rad) |
|-------|-------------------|---------------------|
| 0.058 (old mean) | 0.345 | 1.034 |
| 0.722 (new mean) | 0.028 | 0.083 |
| 0.90 | 0.022 | 0.067 |

The new code provides **12x tighter** position estimates for high-quality observations, enabling the pose estimator to weight good vision data more heavily.

### Selectivity Analysis

The 40% reduction in accepted observations is a feature, not a bug:

1. **Velocity consistency check**: Rejects physically impossible pose jumps
2. **Cross-camera correlation**: Boosts confidence when multiple cameras agree
3. **Weighted tests**: Multiple marginal factors compound to reject borderline observations

The remaining 8,215 observations represent higher-confidence data that should improve overall pose estimation accuracy.

### Input Data Anomaly

A discrepancy was observed in input sample counts:

| Camera | Real Log | Sim Log | Difference |
|--------|----------|---------|------------|
| Camera 0 | 13,484 | 6,192 | -54% |
| Camera 1 | 13,483 | 2,088 | -85% |
| Camera 2 | 13,483 | 6,459 | -52% |
| Camera 3 | 13,483 | 2,446 | -82% |

However, TagIds counts match exactly. This discrepancy is likely due to:
- The `kLoggingDivisor = 2` throttling affecting replay differently
- Timing differences in how periodic() is called during replay
- Input arrays being logged at different frequencies

This does not affect the validity of the score comparison, as both RealOutputs and ReplayOutputs in the sim log process the same inputs.

## Conclusions

### The New VisionFilter Implementation

1. **Produces meaningful scores**: Observations now receive scores proportional to their actual quality (0.0-1.0 range utilized)

2. **Is more selective**: Rejects ~40% of marginal observations that would have been accepted with low confidence

3. **Enables better pose estimation**: Higher scores → tighter standard deviations → pose estimator trusts good vision data more

4. **Differentiates quality**: The score distribution (centered around 0.72) allows the pose estimator to appropriately weight each observation

### Recommendations

1. **Monitor rejection rate**: The 40% rejection rate should be validated in practice. If too many valid observations are rejected, consider:
   - Reducing test weights
   - Adjusting sigmoid midpoints/steepness
   - Relaxing tolerance thresholds

2. **Log individual test scores**: For deeper analysis, enable logging of per-test scores to identify which tests contribute most to rejections

3. **Validate in competition**: Compare pose estimation accuracy between the two approaches using ground truth data if available
